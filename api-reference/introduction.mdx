# API Reference

Complete documentation for all Influship API endpoints.

## Base URL

All API requests are made to:

```
https://api.influship.com
```

## Authentication

All requests require an API key in the `X-API-Key` header:

```bash
curl -H "X-API-Key: YOUR_API_KEY" https://api.influship.com/health
```

## Response Format

All responses are in JSON format with the following structure:

### Success Response

```json
{
  "data": {...},
  "meta": {...}
}
```

### Error Response

```json
{
  "error": {
    "code": "error_code",
    "message": "Human readable error message",
    "details": {...}
  }
}
```

## HTTP Status Codes

- `200` - Success
- `400` - Bad Request
- `401` - Unauthorized
- `404` - Not Found
- `429` - Rate Limited
- `500` - Internal Server Error

## Rate Limits

- **Free tier**: 1,000 requests per hour
- **Pro tier**: 10,000 requests per hour

Rate limit information is included in response headers.

## Billing & Credits

- Responses include billing headers:
  - `X-Credits-Charged`: credits billed for this request
  - `X-Credits-Features`: pricing-impacting features, e.g. `emails,raw`
  - `X-Billing-Plan`: your plan identifier

### Request parameters that affect billing

- **mode**: `lite` (default) | `detailed` – controls richness of normalized fields
- **scope**: `seed_platform` (default) | `all_platforms` – whether to include all known profiles
- **include**: array – feature toggles; currently supports `emails`
- **live**: boolean – when true, returns `202 Accepted` with `job_id` and applies a live multiplier

> Emails are billed as a premium add-on. To request them, pass `include=emails`.

### Example

```http
POST /v1/creators
{ "creator_ids": ["UUID"], "mode": "detailed", "result_scope": "all_platforms", "include": ["emails"] }
X-Credits-Charged: 2.40
X-Credits-Features: emails
X-Billing-Plan: pro
```

## Live Processing

- For heavy operations, set `live=true`
- Behavior:
  - `live=false` (default): returns cached data with 200 if available; may return 202 on cache-miss
  - `live=true`: always returns 202 with `{ job_id }`; results delivered via polling or webhooks
